# プロジェクトルール

本ドキュメントは、このプロジェクトの基本ルールとポリシーを定義します。
すべての貢献者（Claude Code を含む）はこれらのルールに従ってください。

---

## 1. セキュリティファースト

- 設計・実装・レビューのすべての段階でセキュリティを最優先事項として扱う。
- ソースコードにシークレット、認証情報、機密データを埋め込まない。
- 依存ライブラリは最小限にとどめ、採用した各サードパーティライブラリの理由を文書化する（ルール18参照）。
- 依存チェーンを継続的に検証するため、セキュリティスキャンをローカルのクオリティゲートに統合する（ルール20参照）。

## 2. 小さく・集中

- 要件を満たす最小単位を構築し、その後イテレートする。
- 修正は問題にスコープを絞る — 同じ変更で関係のないコードをリファクタリングしない。
- モノリシックな構造より合成（コンポジション）を優先する。

## 3. 関心の分離

- 各モジュール、パッケージ、レイヤーは単一かつ明確な責務を持つ。
- I/O・ビジネスロジック・プレゼンテーションを同一ユニットに混在させない。
- レイヤー間（例: トランスポート、ドメイン、永続化）は明確な境界を定義し、明示的なインターフェース経由で通信する。
- このルールの違反はコードをテストしにくく、推論しにくく、安全に変更しにくいものにする。

## 4. テスト可能な設計

- ユニットを独立してテストできるように設計する（依存性注入、明確なインターフェース）。
- 隠れたグローバル状態を避け、副作用を明示する。
- 関数/メソッドは小さく単一目的に保つ。

## 5. 実装とテストは一緒に

- 実装と同じコミットまたはPRでテストを書く。
- テストされていないプロダクションコードをマージしない。

## 6. ドキュメント必須

- すべてのパブリックAPI、モジュール、非自明な設計決定はドキュメント化しなければならない。
- ドキュメントは `docs/` ディレクトリに置く。インラインコメントは補足であり代替ではない。

## 7. ドキュメントは常に同期

- コードが変更されたとき、対応するドキュメントを同じPRで更新しなければならない。
- 動作変更があるのにドキュメントが更新されていないPRは完成していない。

## 8. 完了前にテスト

- 機能や修正を完了とみなす前に、すべてのテストがローカルで通過しなければならない。
- 変更に関係するテストだけでなく、フルテストスイートを実行する。

## 9. テスト通過後にコミット

- すべてのテストがグリーンになってからのみコミット（またはマージ）する。
- コミットメッセージは何が変わってなぜ変えたかを説明するものでなければならない。

## 10. 大きな変更の回復可能性を保つ

- 大きなまたはリスクのある変更を行う前に、変更前の状態が常に到達可能なよう専用ブランチを作成する。
- 簡単に戻せない動作変更には、フィーチャーフラグや段階的ロールアウトを使用する。
- 破壊的変更が入る前にリリースにタグを打つ。

## 11. ドキュメントとコメントの言語ポリシー

- ソースコードのコメントと主要ドキュメントはすべて**英語**で書く。
- 日本語訳（`docs/ja/`）を並行して維持し、常に同期を保つ。

## 12. コミュニケーション言語

- 貢献者とClaude Code 間のすべてのコミュニケーションは**日本語**で行う。

## 13. 実装前に設計

- プロダクションコードを書く前に、システム全体を見直す：
  1. 高レベルな設計ドキュメントを書く（`docs/design/`）。
  2. フェーズとマイルストーンを含む開発計画を作成する。
  3. 実装を開始する前に明示的な承認を得る。
- 外部API（例: Slack Web API）と連携する場合、**必要なOAuthスコープをすべて**設計段階で列挙し、呼び出すすべてのAPIメソッドと照合する。実行時にスコープ不足を発見するのは防げたエラーである。

## 14. ネイティブコード: Go + Make + クロスコンパイル

- Go はネイティブ/コンパイル済みコードの基本言語。
- ビルドシステム: プロジェクトルートの `Makefile` を使った GNU `make`。
- ターゲットプラットフォーム: `linux/amd64`、`linux/arm64`、`darwin/amd64`、`darwin/arm64`、`windows/amd64`。
  - OS レベルの制約により Windows サポートが困難な場合は、`linux` と `darwin` で可。
- クロスコンパイルは単一ホストマシンから動作しなければならない（`GOOS`/`GOARCH` 変数を使用）。
- コードスタイル: `gofmt` と `golangci-lint` で強制。
- `golangci-lint` はプロジェクトの `go` ディレクティブ以上の Go バージョンでビルドされたものを使う必要があるため、`go install github.com/golangci/golangci-lint/v2/cmd/golangci-lint@latest` でインストールすること。パッケージマネージャー（Homebrew 等）のインストールはバージョンが遅れてバージョン不一致エラーが発生する可能性がある。

## 15. Python: uv 仮想環境

- Python コードは `uv` 管理の仮想環境内で実行しなければならない。
- `pyproject.toml` が正規の設定ファイル。`uv.lock` はコミットする。
- コードスタイル: `ruff`（lint + format）で強制。

## 16. サンドボックス対応のビルド設定

- 開発環境はファイルシステムとネットワークアクセスが制限されたサンドボックス内で動作する。
- ビルドスクリプトは無制限のアウトバウンドネットワークアクセスを前提にしてはならない。必要に応じて依存関係をベンダー化またはキャッシュする。
- ホストレベルの前提条件は `docs/setup.md` に記載する。

## 17. Git と GitHub

- すべてのコードは Git で管理し、権威あるリモートは GitHub。
- ブランチ戦略:
  - `main` — 保護済み。直接プッシュ禁止。
  - `feature/<name>` — 新機能。
  - `fix/<name>` — バグ修正。
  - `docs/<name>` — ドキュメントのみの変更。
  - `chore/<name>` — ツーリング、依存関係の更新。
- `main` へのマージにはプルリクエストが必要。
- PR には何が変わってなぜ変えたかの説明を含める。

## 18. 依存関係管理

- サードパーティの依存関係は本当に必要なときだけ追加する。
- 追加する各依存関係について `docs/dependencies.md` に記載する：
  - 目的と内製しなかった理由。
  - ライセンスとコンプライアンスに関する考慮事項。
- 使用されていない依存関係は速やかに削除する。

## 19. エラーと警告のポリシー

- エラーを無言で無視してはならない（Go では `_ = err` を裸で使わない、Python では `except: pass` を裸で使わない）。
- コンパイラの警告、リンターの警告、テストの警告は未解決のままにしない。エラーとして扱う。
- 一貫したシビリティレベル（`DEBUG`、`INFO`、`WARN`、`ERROR`）で構造化ロギングを使用する。
- 回復可能なエラー（return/log）と回復不能なエラー（明確なメッセージでフェイルファスト）を区別する。

## 20. ローカル自動化によるクオリティゲート（Git フック + Makefile）

- クラウドCIコストを避けるため、クオリティゲートはホスト型CIサービスではなくローカルで強制する。
- `make check` ターゲット（または同等物）は以下を行う必要がある：
  1. コードをリントする。
  2. フルテストスイートを実行する。
  3. 全ターゲットプラットフォーム向けにビルドする。
  4. セキュリティスキャンを実行する（ルール21参照）。
- Git **pre-commit フック**（例: `scripts/hooks/` 配下でリポジトリ管理）はすべてのコミット前に `make check` を実行し、いずれかのステップが失敗するとコミットを拒否する。
- Git **pre-push フック** はリモートへのプッシュ前に同じチェックを実行する。
- フックのインストール方法は `docs/setup.md` に記載し、`make setup` で自動化できる。

## 21. セキュリティスキャン

- Go: `make check` の一部として `govulncheck ./...` を実行する。
- Python: `make check` の一部として `pip-audit`（または `uv run pip-audit`）を実行する。
- マージ前に発見事項を対応する。低リスクとして受け入れる場合は根拠を文書化する。

## 22. バージョニングと変更履歴

- [セマンティックバージョニング](https://semver.org/) (`MAJOR.MINOR.PATCH`) に従う。
- リリースごとに `CHANGELOG.md` を更新して維持する。
- 成果物を配布する前に Git にタグを打つ（`v1.2.3`）。

### リリース成果物のプロセス

1. ワーキングツリーがクリーンであることを確認する（`git status` が変更なしを表示）。これにより `git describe --tags --dirty` がクリーンなバージョン文字列（例: `v1.2.3`）を生成する。
2. `make build-all` を実行して `dist/` 配下に正しいバージョンが埋め込まれたバイナリを生成する。
3. 各バイナリを zip する: `zip scli-<os>-<arch>-<version>.zip scli-<os>-<arch>`。
4. `gh release create <tag> dist/*.zip --title "<tag>" --notes "..."` で GitHub Release を作成する。
5. プッシュのために SSH リモート URL を一時的に HTTPS に切り替えた場合は元に戻す。
